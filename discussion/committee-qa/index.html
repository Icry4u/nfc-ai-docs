    Committee Q&A — Full Preparation Guide¶

    Source

    All answers are based on the submitted Senior Project I report and the actual implemented source code.

    The Project Idea¶

    Why did you choose this idea?

    We noticed that shoppers — especially people with allergies, diabetes, or halal dietary needs — struggle to quickly understand product labels in Saudi supermarkets where packaging mixes Arabic and English. Visually impaired users face an even greater challenge since they cannot read small print at all.

    We built an app where you tap your phone to a product and instantly hear and see its allergen warnings, nutrition info, and AI-powered safe alternatives. It solves a real daily problem that no existing app fully addresses.

    Why a mobile app? Why not a web app?

    The core feature is NFC — the phone physically taps the product tag. NFC hardware only exists on mobile phones, not in browsers. A web app cannot:

    • Access NFC hardware
    • Work offline in a supermarket with poor WiFi
    • Send vibration haptic feedback to the user

    The entire interaction model requires a phone in the user's hand.

    Are there similar ideas out there?

    Yes — there are apps that partially overlap with what we built. We identified 3 close existing apps:

    Yuka is a popular food scanning app available on iOS and Android. Users scan a product barcode with the camera and receive a health score, nutrition breakdown, and ingredient analysis.

    • Uses barcode scanning (camera-based), not NFC
    • Provides nutrition scores and ingredient flags
    • No AI-generated alternative product recommendations
    • No Arabic language support
    • No Arabic/English bilingual support
    • No text-to-speech for visually impaired users

    Open Food Facts is an open-source community-driven food database app. Users scan barcodes to view crowdsourced nutrition and ingredient data.

    • Uses barcode scanning (camera-based), not NFC
    • Community-sourced data — accuracy varies
    • No personalized allergen profiles
    • No AI recommendations
    • Limited Arabic support
    • No text-to-speech

    The Carrefour KSA app allows shoppers to scan products in-store for pricing and basic product info.

    • Uses barcode scanning, not NFC
    • Focused on price and inventory, not nutrition safety
    • No allergen warnings or dietary filtering
    • No AI recommendations
    • No text-to-speech

    How many did we find? We found 3 apps that are close or similar in concept, plus reviewed over 20 academic research papers on related systems.

    How does your project differ from them?
    Feature Our App Yuka Open Food Facts Carrefour KSA
    Scanning method NFC tap Barcode (camera) Barcode (camera) Barcode (camera)
    AI recommendations ✅ 3 safe alternatives ❌ ❌ ❌
    Allergen profile filtering ✅ Personal profile Partial Partial ❌
    Arabic language ✅ Full bilingual ❌ ❌ Partial
    Text-to-speech ✅ Arabic + English ❌ ❌ ❌
    Works offline ✅ Local database Partial Partial ❌
    Visually impaired support ✅ ❌ ❌ ❌

    The key differentiator is NFC + AI + Accessibility combined in one app. No existing solution targets visually impaired shoppers in Saudi Arabia with an AI-powered, Arabic-speaking, NFC-based assistant.

    What features does your project have that are not in the market yet?
    1. NFC tap-to-scan — no camera or barcode needed, instant product identification
    2. AI-powered safe alternatives — LLaMA 3.3 70B (via Groq) analyzes your personal allergen profile and recommends 3 specific safe products with reasons and nutrition data
    3. Arabic text-to-speech — reads results aloud to support visually impaired users
    4. Full bilingual product data — Arabic with diacritical marks (tashkeel) + English in the same database
    5. Personal allergen profile — user sets their restrictions once (dairy-free, gluten-free, nut allergy, vegan) and every recommendation is filtered against that profile
    Does your project solve a real life problem?

    Yes. Misreading a food label can cause a life-threatening allergic reaction. Visually impaired users cannot read labels at all. Our app removes these risks instantly at the point of purchase.

    What category does your project fall under?

    Health and consumer safety — with elements of AI/machine learning, accessibility technology, and retail technology.

    Database¶

    How many databases do you have?

    One — a local Room database (built on SQLite) named nfcai_shopping_db.

    Room is Android's official database library that wraps SQLite with compile-time SQL verification, LiveData integration, and DAO (Data Access Object) patterns.

    How many tables do you have?

    Two tables:

    Stores all product information

    Column Type Description
    id INTEGER (PK) Auto-generated primary key
    nfcTagId TEXT (UNIQUE) NFC chip hardware ID
    name TEXT Product name in English
    nameAr TEXT Product name in Arabic
    brand TEXT Brand name
    allergens TEXT Comma-separated allergen list
    allergensAr TEXT Allergens in Arabic
    calories REAL Per 100g
    protein REAL Per 100g
    carbs REAL Per 100g
    fat REAL Per 100g
    weightGrams INTEGER Package weight
    ingredients TEXT Ingredients (English)
    ingredientsAr TEXT Ingredients (Arabic)
    isHalal INTEGER 1 = halal, 0 = not halal
    imageEmoji TEXT Emoji icon for display

    Records every product the user has scanned

    Column Type Description
    id INTEGER (PK) Auto-generated primary key
    productId INTEGER (FK) References products.id
    scannedAt INTEGER Unix timestamp in milliseconds

    Relationship: One product → many scan records (One-to-Many). Cascade delete — removing a product automatically removes its scan history.

    Why did you choose that database?

    We chose Room (SQLite) because:

    • It is Android's official recommended local database
    • Works fully offline — no internet needed after setup
    • Integrates with LiveData for automatic UI updates when data changes
    • Lightweight enough to run on devices with 2GB RAM without performance issues
    • Provides compile-time SQL verification — SQL errors are caught at build time, not at runtime
    Was it difficult to deal with?

    The main challenge was designing a bilingual schema — every product needs both Arabic and English fields with diacritical marks (tashkeel). We also pre-seeded 15 real Saudi market products (Almarai, Weetabix, Galaxy, KDD, etc.) with verified nutrition data.

    Managing database schema changes (migrations) was also a learning point. We used fallbackToDestructiveMigration() during development which rebuilds the database when the schema changes.

    What type of normalization did you apply?

    We applied Third Normal Form (3NF):

    Normal Form How We Applied It
    1NF Every field holds one atomic value — no repeating groups
    2NF No partial dependencies — scan_history only stores productId and timestamp, never product details
    3NF No transitive dependencies — nutrition data lives only in products, never duplicated in scan_history

    This eliminates data redundancy: updating a product's allergen info only requires changing one row in the products table.

    Is it relational (SQL) or non-relational (NoSQL)?

    Relational (SQL) — Room/SQLite. The two tables have a foreign key relationship:

    scan_history.productId → products.id

    This enforces data integrity: you cannot have a scan record pointing to a product that doesn't exist.

    Why One-to-Many?

    • One product can be scanned many times (a user might check Almarai Milk repeatedly over weeks)
    • Each scan record belongs to exactly one product
    • It would be wrong to use Many-to-Many here because a single scan event is always for one specific product at one point in time

    Methodology¶

    What model did you use and why?

    We used Agile (Incremental Development).

    Our project had evolving requirements — we added text-to-speech, then bilingual support, then the AI recommendation engine as we discovered what worked during testing. Agile suited us because:

    • It allowed us to build the app in small functional increments (NFC scanning → database → AI → TTS)
    • Each piece was tested before moving forward
    • It aligned with our 12-week academic schedule divided into 4 sprints
    • It supports collaboration between team members on different features

    We considered Waterfall but rejected it because our requirements were not fully fixed from the start — new insights from testing changed our design decisions throughout development.

    Abstract & Conclusion¶

    What is your project about?

    NFC-AI Smart Shopping Assistant combines Near Field Communication (NFC) and Artificial Intelligence to help shoppers — especially those with allergies, dietary restrictions, or visual impairments — make safe product choices.

    Users tap their phone to an NFC-tagged product and instantly receive:

    • Allergen warnings
    • Full nutrition information
    • AI-generated safe alternative products
    • Text-to-speech audio — all in Arabic and English
    What is your conclusion?

    Senior Project I established the research, requirements, and design foundation. In Senior Project II we fully implemented the system.

    We evaluated and rejected several alternatives:

    Alternative Considered Why Rejected
    On-device AI model Phone hardware not powerful enough; storage too large
    Barcode scanning Excluded to keep NFC as the primary interaction
    IoT integration (smart fridges) Scope too large for this phase

    The result is a focused, working app that delivers on all three core goals: accessibility, allergen safety, and AI-powered recommendations.

    Doing the Project¶

    What problems did you face during your work?

    1. NFC tag formats — Different NFC tags (NDEF vs raw) behaved differently across Android devices. We had to handle multiple tag types in the intent filter configuration.

    2. AI hallucinations — The AI sometimes returned product names in Chinese or Japanese characters. We built a filter that blocks any response where more than 10% of characters are outside Arabic/Latin ranges.

    3. Bilingual text layout — Arabic is right-to-left, English is left-to-right. Mixing both in the same screen required careful BiDi (bidirectional) text configuration using android:textDirection="locale".

    4. API key security — We learned that API keys embedded in Android source code are extractable even from the compiled APK binary. This is a known limitation we plan to fix in a future backend proxy.

    What was the most challenging part?

    The AI recommendation engine. We had to carefully engineer the prompt so the AI:

    • Understands the user's exact allergen profile
    • Returns structured JSON in both languages
    • Recommends the same type of product (not switching categories)
    • Detects hidden allergens (e.g., dairy hidden in chocolate, gluten in soy sauce)
    • Never hallucinates dangerous recommendations for someone with a life-threatening allergy
    What are you planning for future development?
    • Move the API key to a secure backend proxy so it is never bundled in the APK
    • Add barcode scanning as a fallback when NFC tags are not present
    • Explore on-device AI models as phone processors improve (no cloud dependency)
    • Expand product database via supermarket API integration
    • Add SQLCipher for encrypted local database storage
    • Integrate with IoT devices for home inventory tracking

    Show me where you connect to the database

    The database is created in AppDatabase.java using the Singleton pattern to ensure only one instance exists throughout the app's lifecycle:

    AppDatabase.java — Singleton Database Instance
    @Database(entities = {Product.class, ScanHistory.class}, version = 5, exportSchema = false)
public abstract class AppDatabase extends RoomDatabase {

    private static volatile AppDatabase INSTANCE;

    public abstract ProductDao productDao();
    public abstract ScanHistoryDao scanHistoryDao();

    public static AppDatabase getInstance(Context context) {
        if (INSTANCE == null) {
            synchronized (AppDatabase.class) {
                if (INSTANCE == null) {
                    INSTANCE = Room.databaseBuilder(
                                    context.getApplicationContext(),
                                    AppDatabase.class,
                                    "nfcai_shopping_db")   // (1)
                            .fallbackToDestructiveMigration() // (2)
                            .addCallback(seedDatabaseCallback) // (3)
                            .build();
                }
            }
        }
        return INSTANCE;
    }
}
    1. Database file name on disk: nfcai_shopping_db
    2. If schema changes, wipe and rebuild (development mode)
    3. Pre-seed 15 products on first creation

    How data is queried — using a DAO (Data Access Object). When an NFC tag is scanned, its hardware ID is used to look up the product:

    ProductDao.java — NFC Tag Lookup
    @Dao
public interface ProductDao {

    // Called when NFC tag is tapped — finds product by hardware tag ID
    @Query("SELECT * FROM products WHERE nfcTagId = :tagId LIMIT 1")
    Product getProductByTagIdSync(String tagId);

    // Used by HomeFragment to show recent scans live
    @Query("SELECT * FROM products")
    LiveData<List<Product>> getAllProducts();

    // Used to skip seed if products already exist
    @Query("SELECT COUNT(*) FROM products")
    int getProductCount();

    // Manual search by product name (Arabic or English)
    @Query("SELECT * FROM products WHERE name LIKE '%' || :query || '%' " +
           "OR nameAr LIKE '%' || :query || '%' LIMIT 1")
    Product searchByNameSync(String query);
}

    Show me the AI connection code

    The AI call is made in GroqClient.java. It sends the product details + user's allergen profile to Groq's API and receives 3 safe alternative recommendations in JSON format:

    GroqClient.java — Sending the AI Request
    private static final String API_URL = "https://api.groq.com/openai/v1/chat/completions";
private static final String MODEL   = "llama-3.3-70b-versatile";

public void getRecommendations(String productName, String ingredients,
                               String allergens, String userProfile,
                               String language, float calories, float fat,
                               GroqCallback callback) {
    try {
        String prompt = buildPrompt(productName, ingredients,
                                    allergens, userProfile,
                                    language, calories, fat); // (1)

        JSONObject systemMsg = new JSONObject();
        systemMsg.put("role", "system");
        systemMsg.put("content",
            "You are a food safety assistant. Verify EACH recommendation " +
            "is safe for the customer's FULL dietary profile. A wrong suggestion " +
            "could cause a dangerous allergic reaction.");

        JSONObject body = new JSONObject();
        body.put("model", MODEL);
        body.put("messages", messages);
        body.put("temperature", 0.1);   // (2)
        body.put("max_tokens", 512);

        Request request = new Request.Builder()
                .url(API_URL)
                .addHeader("Authorization", "Bearer " + API_KEY)
                .addHeader("Content-Type", "application/json")
                .post(RequestBody.create(body.toString(), JSON))
                .build();

        try (Response response = client.newCall(request).execute()) { // (3)
            String responseBody = response.body().string();
            List<Recommendation> recs = parseResponse(responseBody);
            callback.onSuccess(recs);
        }

    } catch (Exception e) {
        callback.onError(e.getMessage());
    }
}
    1. buildPrompt() creates a detailed instruction with allergen rules, product type matching rules, and JSON format requirements
    2. Temperature 0.1 = very low creativity, more predictable/safe output
    3. OkHttp3 executes the HTTP request with 15s connect timeout, 30s read timeout

    The response is then parsed and sanitized:

    GroqClient.java — Parsing and Sanitizing the AI Response
    private List<Recommendation> parseResponse(String responseBody) throws Exception {
    JSONObject root = new JSONObject(responseBody);
    String content = root
            .getJSONArray("choices")
            .getJSONObject(0)
            .getJSONObject("message")
            .getString("content").trim();

    // Strip markdown code fences if the model added them
    if (content.startsWith("```")) {
        content = content.replaceAll("```[a-z]*\\n?", "").replace("```", "").trim();
    }

    JSONArray arr = new JSONArray(content);
    List<Recommendation> recs = new ArrayList<>();

    for (int i = 0; i < arr.length(); i++) {
        JSONObject obj = arr.getJSONObject(i);
        String name   = obj.optString("name", "");
        String nameAr = sanitize(obj.optString("nameAr", "")); // (1)
        String reason = sanitize(obj.optString("reason", ""));

        if (!name.isEmpty() && !hasGibberish(name) && !hasGibberish(nameAr)) { // (2)
            Recommendation rec = new Recommendation(name, nameAr, reason,
                                                    obj.optInt("matchPercent", 80));
            rec.calories = (float) obj.optDouble("calories", 0);
            rec.protein  = (float) obj.optDouble("protein",  0);
            // ...
            recs.add(rec);
        }
    }
    return recs;
}
    1. sanitize() strips any character outside Arabic, Latin, digits, and basic punctuation
    2. hasGibberish() rejects the recommendation if >10% of characters are non-Arabic/non-Latin (blocks Chinese/Japanese injection)

    Security¶

    How did you make your app secure?

    All AI text responses are cleaned before displaying to the user:

    GroqClient.java — sanitize()
    /**
 * Remove characters outside Arabic, English, numbers, and basic punctuation.
 * Prevents unexpected scripts from appearing in the UI.
 */
private String sanitize(String text) {
    if (text == null) return "";
    return text.replaceAll(
        "[^\\u0600-\\u06FF\\u0020-\\u007EA-Za-z0-9%'\"().,،؛:-]", ""
    ).trim();
}

    This regex keeps only:

    • \u0600-\u06FF — Arabic script
    • \u0020-\u007E — Basic Latin (ASCII printable)
    • Standard punctuation

    Blocks AI responses that contain unexpected character scripts (Chinese, Japanese, Korean, etc.):

    GroqClient.java — hasGibberish()
    /**
 * Detect gibberish — Chinese, Japanese, Korean, or other unexpected scripts.
 * If more than 10% of characters are outside Arabic/Latin, the recommendation
 * is discarded entirely.
 */
private boolean hasGibberish(String text) {
    if (text == null || text.isEmpty()) return false;
    int bad = 0;
    for (int i = 0; i < text.length(); i++) {
        char c = text.charAt(i);
        boolean ok = (c >= 0x0600 && c <= 0x06FF)   // Arabic
                  || (c >= 0x0020 && c <= 0x007E)    // Basic Latin
                  || (c >= 0xFB50 && c <= 0xFDFF)    // Arabic Pres. Forms-A
                  || (c >= 0xFE70 && c <= 0xFEFF)    // Arabic Pres. Forms-B
                  || Character.isWhitespace(c);
        if (!ok) bad++;
    }
    return bad > text.length() * 0.1; // More than 10% bad = gibberish
}

    Testing¶

    How did you test your project?

    Tested on a real Android device — every screen, NFC scan, AI response, and edge case:

    • Unknown NFC tag (tag not in database)
    • No internet connection (AI call fails gracefully)
    • Empty allergen profile (app suggests general alternatives)
    • Unreadable NFC tag (corrupted or unsupported format)
    • AI response with unexpected characters (gibberish filter triggered)

    End-to-end flow tested continuously:

    NFC tap → Hardware ID read → Database lookup → Product found
     → AI request sent → Response parsed → Sanitized → Displayed
     → TTS reads result aloud

    Individual functions tested in isolation:

    • sanitize() — tested with Arabic, English, Chinese, emoji, and mixed inputs
    • hasGibberish() — tested with valid Arabic/English and Chinese character strings
    • JSON response parser — tested with malformed JSON, missing fields, empty arrays
    What is the testing approach, level, and type?
    Dimension Answer
    Approach Black-box (user behavior) + White-box (code paths)
    Level Unit (functions) + System (full app on device)
    Type Functional + Boundary testing
    Internal or external? How long did testing take?

    Tested internally by the development team throughout development in an Agile manner — each feature was tested before moving to the next sprint.

    Bug fixes were typically resolved within 1–3 days per issue within the same sprint. The iterative nature of Agile meant testing was continuous, not a single final phase.
